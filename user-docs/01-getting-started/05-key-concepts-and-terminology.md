# Key concepts and terminology

This glossary defines the core terms used throughout FlowForge. Understanding these concepts will help you navigate the platform and follow the documentation more easily.

## Core concepts

### Project

A top-level container that holds all your API testing work — specs, ideas, flows, scenarios, connections, variables, and settings. Each project is isolated with its own team members and role-based permissions.

### Version folder

A folder within a project that groups specs and flows by API version (e.g., "v3", "v2"). Version folders help you manage testing across different API releases. Each version folder can have its own API connection and configuration.

## Spec Manager terminology

### Spec (specification)

A Markdown file describing a single API endpoint — its URL, HTTP method, request parameters, request body schema, response codes, and response body schema. Specs are imported from OpenAPI/Swagger files and serve as the source of truth for AI-powered test generation.

### OpenAPI / Swagger

Industry-standard formats for describing REST APIs. FlowForge supports both OpenAPI 3.x and Swagger 2.0 specs. When imported, FlowForge automatically splits the spec into individual per-endpoint files.

### Distilled spec

A condensed version of a spec file (~50-100 lines) optimized for AI consumption. Distilled specs are auto-generated during import and used by the AI when generating flows. You'll see them in the file tree under `_system/_distilled/` folders.

### Digest

A lightweight index (~2-3 lines per endpoint) of all endpoints in a version folder. Used by AI during idea generation when there are many endpoints, to stay within context limits.

### Idea

A test scenario suggestion generated by AI from your API specs. An idea describes a multi-step workflow in natural language — for example, "Create a category, then create an article in that category, retrieve the article, and clean up both." Ideas are the starting point for flow generation.

### Flow

An XML document that defines a complete test scenario — the exact sequence of API calls, what data to capture between steps, and what assertions to check. Flows are authored in the Spec Manager and can be:
- **AI-generated** from ideas
- **Written manually** in the XML editor
- **Refined** through the Flow Designer chat

> **Terminology note:** The term "flow" is used in the Spec Manager context (the authoring artifact). Once a flow is registered for execution, it becomes a "scenario" in the Scenario Manager.

### Flow XML

The XML format used to define flows. A flow XML file contains:
- **Metadata** — Name, description, and configuration
- **Steps** — Ordered API calls with method, path, headers, and body
- **Assertions** — Expected outcomes to verify (status codes, response field values)
- **Captures** — Values extracted from responses for use in later steps

### API rules

Per-version-folder configuration that gets injected into all AI prompts. Rules let you tell the AI about API-specific behaviors, conventions, or constraints that aren't captured in the spec — for example, "always include `workspace_id` in POST requests" or "the `status` field returns integers, not strings."

### Diagnostic lessons

Auto-learned knowledge captured when AI diagnosis successfully fixes a test failure. Lessons are stored per version folder and automatically included in future AI operations, so the same mistake isn't repeated.

## Scenario Manager terminology

### Scenario

A runnable test instance created from a flow. Scenarios appear in the Scenario Manager's tree view, organized by version and folder. A scenario includes the flow definition plus its execution context (connection, variables, overrides).

### Step

A single API call within a scenario. Each step specifies an HTTP method, endpoint path, request body, and expected outcomes. Steps execute sequentially, and data can flow from one step to the next through captures.

### Assertion

A check that verifies an expected outcome within a step. Assertions can validate:
- **Status code** — The HTTP response status (e.g., 200, 201, 404)
- **Response fields** — Specific values in the JSON response body (e.g., `response.data.name` equals "Test")
- **Existence** — Whether a field is present or absent in the response

### Capture

A mechanism to extract a value from a step's response and save it for use in later steps. For example, capturing the `id` field from a "Create Article" response so it can be used in subsequent "Get Article" and "Delete Article" steps. Captures use the syntax `{{state.variableName}}` to reference saved values.

### Run

A single execution of one or more scenarios against a live API endpoint. Each run records:
- Start and end timestamps
- Pass/fail status per step
- Full request/response data for each step
- Assertion outcomes

### Breakpoint

A pause marker set on a specific step. When the test runner reaches a breakpoint, execution pauses so you can inspect the current state before continuing. Useful for debugging multi-step scenarios.

## Connections & variables

### Connection

A saved configuration for connecting to an API endpoint. A connection includes:
- **Base URL** — The API's root URL (e.g., `https://api.example.com`)
- **API version** — The version path segment (e.g., `v2`)
- **Auth provider** — The authentication method (OAuth 2.0, bearer token, API key, basic auth, cookie)
- **Credentials** — The actual secret values (stored server-side, never exposed to the browser)

### Project variable

A key-value pair defined at the project level and available across all flows. Referenced in flow XML using `{{proj.variableName}}` syntax. Common uses include workspace IDs, default user emails, or environment-specific values.

### State variable

A variable set during test execution through captures. Referenced using `{{state.variableName}}` syntax. State variables only exist within a single run — they're created by one step and consumed by later steps.

### Context variable

Runtime variables provided by the execution environment, referenced using `{{ctx.variableName}}` syntax. These include system-provided values like timestamps or run identifiers.

## Variable syntax

All variable references in flow XML use double curly braces (mustache syntax):

| Prefix | Example | Source |
|---|---|---|
| `proj.` | `{{proj.workspace_id}}` | Project variables (Settings > Variables) |
| `state.` | `{{state.article_id}}` | Captured from previous step responses |
| `ctx.` | `{{ctx.timestamp}}` | Runtime context values |

## AI features

### AI model

The Claude language model used for AI operations (idea generation, flow generation, chat, diagnosis). FlowForge defaults to Claude Sonnet 4.6 but allows you to select different models. Model selection affects both quality and cost.

### AI credits

A budget system that limits AI usage per project. Each AI operation consumes credits based on the number of tokens processed. When credits are exhausted, AI features are temporarily disabled until the budget is increased.

### AI diagnosis

An automated analysis tool that examines a failed test step, compares the actual response against the API spec, identifies the root cause, and suggests (or automatically applies) a fix. Available from the Diagnose tab when viewing a failed step.

## Public API

### API key

A secret token used to authenticate requests to the FlowForge Public API. API keys are project-scoped and managed in Settings > API Keys. They allow external tools and CI/CD pipelines to trigger test runs programmatically.
